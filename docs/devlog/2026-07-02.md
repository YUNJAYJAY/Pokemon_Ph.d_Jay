# 2026-07-02: 포켓몬 박사 챗봇 프로젝트 정리

## 1. 프로젝트 목표

본 프로젝트의 최종 목표는 포켓몬 박사님 역할을 수행하는 AI 챗봇을 만드는 것이다.

단순히 포켓몬의 이름이나 기본 정보를 알려주는 수준을 넘어서, 다음과 같은 기능을 제공하는 것을 목표로 한다.

- 포켓몬 종류 및 기본 정보 제공
- 포켓몬 타입 상성 설명
- 기술 효과 및 배틀 판정 설명
- 세대별 타입 상성 및 규칙 차이 반영
- 각 게임별 공략 정보 제공
- 사용자의 질문에 대해 한국어로 자연스럽게 답변
- 포켓몬 배틀 상황에서 유리한 선택지를 추론하여 설명

이를 위해 LLM, RAG, 규칙 엔진, 파인튜닝을 결합한 구조를 사용한다.

## 2. 모델 선정 검토

포켓몬 박사 챗봇에 사용할 수 있는 LLM 후보를 비교하였다.

| 순위 | 모델 | 추천도 | 선정 이유 |
| --- | --- | --- | --- |
| 1 | Qwen3-8B / Qwen3-4B | 매우 높음 | 한국어 처리, 추론 능력, 파인튜닝, 개인 배포 측면에서 균형이 좋음 |
| 2 | Gemma 3 / Gemma 4 계열 | 높음 | 비교적 가볍고 멀티모달 확장 가능성이 있음. 단, 라이선스와 생태계 확인 필요 |
| 3 | Mistral Small 3.2 24B | 중간 | 성능은 좋지만 개인 프로젝트에서 배포하기에는 다소 무거움 |
| 4 | Phi-4-mini | 중간 | 매우 가볍지만 한국어 기반 챗봇에서는 Qwen 계열보다 아쉬움 |

## 3. Qwen3 모델 검토

Qwen3는 dense 모델과 MoE 모델을 포함하는 LLM 계열이다. 본 프로젝트에서는 개인 개발 및 배포 편의성을 고려하여 dense 계열을 우선 검토하였다.

### 3.1 Dense 모델

Dense 모델은 가장 일반적인 LLM 구조이다.

사용자의 질문이 입력되면 모델 내부의 대부분의 파라미터가 함께 사용된다. 예를 들어 8B dense 모델의 경우, 추론 시 약 80억 개 규모의 파라미터가 전체적으로 동작한다.

대표적인 dense 모델에는 다음이 있다.

- Qwen 8B
- Llama 8B
- Mistral 7B

Dense 모델의 장점은 다음과 같다.

- 구조가 단순함
- 파인튜닝 자료와 사례가 많음
- 로컬 배포가 비교적 쉬움
- 개인 프로젝트에서 다루기 적합함

반면, 모델 크기가 커질수록 추론 비용과 메모리 사용량이 증가한다는 단점이 있다.

### 3.2 MoE 모델

MoE는 Mixture of Experts의 약자로, 전문가 혼합 모델을 의미한다.

MoE 모델은 내부에 여러 전문가 모델을 가지고 있으며, 질문이 들어오면 그중 일부 전문가만 선택하여 사용한다. 예를 들어 전체 파라미터가 30B 규모인 모델이라도 실제 질문 처리에는 3~6B 정도의 파라미터만 활성화될 수 있다.

MoE 모델의 장점은 다음과 같다.

- 전체 모델 규모에 비해 추론 효율이 좋음
- 다양한 작업을 전문가 단위로 처리할 수 있음
- 대형 모델의 성능과 비교적 낮은 추론 비용을 함께 기대할 수 있음

하지만 개인 프로젝트에서는 다음과 같은 어려움이 있다.

- 구조가 복잡함
- 파인튜닝이 dense 모델보다 까다로움
- 배포 및 최적화 난이도가 높음

따라서 초기 개발 단계에서는 MoE보다 dense 모델을 사용하는 것이 적합하다고 판단하였다.

## 4. Qwen3의 장점

Qwen3 계열은 다음 세 가지 측면에서 본 프로젝트에 적합하다.

### 4.1 Reasoning

Reasoning은 추론 능력을 의미한다.

포켓몬 챗봇에서는 단순 정보 제공뿐만 아니라, 배틀 상황에서 어떤 선택이 더 유리한지 설명할 수 있어야 한다.

예를 들어 사용자가 특정 포켓몬 간 대결을 질문했을 때, 챗봇은 단순히 타입 상성만 말하는 것이 아니라 다음과 같이 판단해야 한다.

- 어떤 타입 공격이 효과적인지
- 어떤 포켓몬이 방어적으로 유리한지
- 어떤 기술 선택이 더 적합한지
- 세대별 규칙 차이가 있는지
- 전체적으로 어떤 포켓몬이 더 유리한지

따라서 추론 능력은 포켓몬 박사 챗봇에서 중요한 요소이다.

### 4.2 Instruction Following

Instruction following은 사용자의 지시를 얼마나 정확히 따르는지를 의미한다.

포켓몬 데이터는 세대, 게임 버전, 타입, 기술, 특성 등에 따라 정보가 달라질 수 있다. 이 능력이 부족하면 사용자가 5세대 기준으로 질문했는데 다른 세대 기준으로 답변하는 문제가 발생할 수 있다.

따라서 챗봇은 사용자가 지정한 조건을 정확히 유지하며 답변할 수 있어야 한다.

### 4.3 Multilingual Support

포켓몬 정보는 한국어, 영어, 일본어 이름이 함께 사용되는 경우가 많다.

예를 들어 한국어 사용자는 “이상해씨”라고 질문할 수 있지만, 데이터베이스나 API에서는 “bulbasaur”라는 영어 이름으로 저장되어 있을 수 있다.

따라서 다음과 같은 다국어 처리 능력이 필요하다.

- 한국어 포켓몬 이름 인식
- 영어 포켓몬 이름 매핑
- 일본어 명칭 확장 가능성
- 한국어 자연어 답변 생성

이러한 이유로 multilingual support는 본 프로젝트에서 필수적인 요소이다.

## 5. 모델 사용 전략

프로젝트 단계별 모델 사용 전략은 다음과 같다.

| 단계 | 모델 | 용도 |
| --- | --- | --- |
| 시작 모델 | Qwen3-4B-Instruct-2507 | 초기 개발, 테스트, 로컬 실험 |
| 서비스 모델 v1 | Qwen3-8B-Instruct | 실제 서비스용 기본 모델 |

Qwen3-14B는 기본적으로 32K 컨텍스트를 지원하며, YaRN 사용 시 131K까지 확장 검증된 것으로 알려져 있다.

Qwen3-4B-Instruct-2507은 4B 규모이면서 262,144 토큰 컨텍스트를 명시하고 있어, 초기 실험 및 긴 문서 기반 질의응답에 적합한 후보로 볼 수 있다.

## 6. 최종 서비스 구조

최종적으로 목표하는 서비스 구조는 다음과 같다.

```text
Qwen3-8B + QLoRA + RAG + 규칙 엔진
```

각 구성 요소의 역할은 다음과 같다.

### 6.1 RAG

RAG는 Retrieval-Augmented Generation의 약자로, 검색 증강 생성을 의미한다.

일반적인 LLM은 학습 시점 이후의 정보나 세부적인 게임 데이터를 틀리게 답변할 수 있다. 따라서 포켓몬 데이터베이스에서 관련 정보를 검색한 뒤, 그 정보를 바탕으로 답변하게 만드는 방식이 필요하다.

RAG를 사용하면 다음과 같은 장점이 있다.

- 최신 데이터 반영이 쉬움
- 세대별 정보 관리가 쉬움
- LLM의 환각 현상을 줄일 수 있음
- 데이터베이스만 수정해도 정보 업데이트 가능
- 게임별 공략 문서 확장에 유리함

### 6.2 QLoRA

QLoRA는 LLM을 비교적 가볍게 파인튜닝하는 방법이다.

기본이 되는 LoRA는 Low-Rank Adaptation의 약자로, 전체 모델을 모두 학습시키지 않고 작은 추가 가중치만 학습하는 방식이다.

8B 모델 전체를 학습시키려면 많은 GPU 메모리가 필요하다. 따라서 모델 본체는 대부분 고정하고, 작은 어댑터만 학습하는 방식이 효율적이다.

QLoRA는 LoRA에 양자화를 결합한 방식이다. 양자화를 통해 모델의 메모리 사용량을 줄이고, 개인 개발 환경에서도 파인튜닝을 시도할 수 있도록 한다.

본 프로젝트에서는 QLoRA를 통해 다음을 학습시키는 것을 목표로 한다.

- 포켓몬 박사님 같은 답변 말투
- 답변 형식 통일
- 상성 설명 방식
- 게임 공략형 응답 스타일
- 한국어 기반 질의응답 패턴

### 6.3 규칙 엔진

타입 상성이나 배틀 판정처럼 명확한 규칙이 존재하는 부분은 LLM에게만 맡기지 않고 규칙 엔진으로 처리한다.

예를 들어 다음과 같은 정보는 규칙 엔진에서 계산하는 것이 적합하다.

- 불꽃 타입이 풀 타입에게 2배 피해
- 노말 타입 기술이 고스트 타입에게 무효
- 세대별 타입 상성 차이
- 복합 타입 포켓몬의 최종 배율 계산
- 기술의 명중률, 위력, 타입, 분류

LLM은 규칙 엔진의 결과를 바탕으로 자연스럽게 설명하는 역할을 맡는다.

## 7. 전체 개발 Workflow

프로젝트의 전체 개발 흐름은 다음과 같다.

1. 포켓몬 데이터베이스 구축
2. 타입 상성 규칙 엔진 구현
3. 웹 시각화 기능 구현
4. LLM 연결
5. RAG 기반 정보 검색 구조 구축
6. QLoRA를 활용한 말투 및 답변 형식 파인튜닝

초기에는 데이터 수집과 정제가 가장 중요하다. 이후 규칙 엔진과 RAG 문서를 구축하고, 마지막 단계에서 LLM 파인튜닝을 진행한다.
